# File map — what every file/folder is responsible for

Updated whenever files are added, moved, or repurposed. Template/boilerplate files that we
have not modified are summarised per-folder rather than listed one by one.

```
Ogbaje/
├── ARCHITECTURE.md          top-level architecture overview + build status
├── docs/
│   ├── STACK.md             stack choices and reasoning
│   ├── DEPENDENCIES.md      every dependency: version, health, why (incl. rejected ones)
│   ├── DATA-MODEL.md        Firestore collections/shapes, contact linking, access matrix
│   ├── FLOWS.md             text diagrams of all significant logic
│   ├── FILE-MAP.md          ← this file
│   └── SETUP.md             machine + Firebase setup steps (toolchain gaps flagged)
│
├── README.md                project overview + repo map
├── firestore.rules          Firestore security rules (owner-writes, recipient-reads;
│                            demo relaxation documented inline)
├── storage.rules            Storage rules for alert photos
│
├── OgbajeApp/               Expo SDK 57 app (TypeScript), Android-first, development build
│   ├── app.json             Expo config: android.package com.ogbaje.app, googleServicesFile,
│   │                        all Android permissions, and config plugins (@react-native-
│   │                        firebase/app, expo-location, expo-image-picker, expo-notifications,
│   │                        expo-build-properties)
│   ├── eas.json             EAS Build profiles (development/preview/production APK/AAB) —
│   │                        cloud builds so no local Android SDK is strictly required
│   ├── google-services.json ← user-provided, GIT-IGNORED (docs/SETUP.md)
│   ├── index.ts             Expo entry (registerRootComponent → App)
│   ├── modules/ogbaje-sms/  OUR local Expo Module — the SMS failsafe
│   │   ├── expo-module.config.json   declares the Android module for autolinking
│   │   ├── index.ts         typed JS face (requireOptionalNativeModule('OgbajeSms'))
│   │   └── android/
│   │       ├── build.gradle           applies expo-module-gradle-plugin
│   │       └── src/main/
│   │           ├── AndroidManifest.xml  declares SEND_SMS
│   │           └── java/expo/modules/ogbajesms/OgbajeSmsModule.kt
│   │                        Kotlin: direct SIM SMS via SmsManager, multipart-aware,
│   │                        send-result callbacks (all RN SMS packages are dead — DEPS.md)
│   ├── android/  ios/       GENERATED by `expo prebuild` (git-ignored); not committed
│   ├── src/
│   │   ├── config.ts        non-secret config (dashboard base URL)
│   │   ├── theme.ts         colors/spacing/radius (red = SOS only)
│   │   ├── types.ts         TS mirrors of docs/DATA-MODEL.md
│   │   ├── firebase/
│   │   │   ├── users.ts     profile, contacts CRUD, phone normalisation (+234),
│   │   │   │                registered-user lookup, linkedUid backfill, FCM token
│   │   │   └── alerts.ts    createAlert, appendLocation, timeline events, photo docs,
│   │   │                    mode switch, resolve, watchers
│   │   ├── services/
│   │   │   ├── alertEngine.ts        THE master trigger flow (FLOWS.md §1); idempotent
│   │   │   │                         under panic-mashing; streaming + mode listeners
│   │   │   ├── locationService.ts    time-budgeted getFix with cached fallback; streaming
│   │   │   ├── smsService.ts         JS bridge to the local OgbajeSms Expo module + composer
│   │   │   ├── connectivity.ts       NetInfo wrapper: isOnline snapshot + reconnect events
│   │   │   ├── photoQueue.ts         offline photo queue-and-sync (FLOWS.md §2)
│   │   │   ├── checkinService.ts     dead-man's switch state + deadline evaluation (§3)
│   │   │   ├── batteryWatch.ts       critical-battery evaluation, no-refire latch (§4)
│   │   │   ├── foregroundService.ts  react-native-background-actions loop: 60s tick
│   │   │   │                         evaluating check-in + battery watchers
│   │   │   ├── incomingAlerts.ts     receive side: realtime listener for alerts where
│   │   │   │                         I'm a recipient + contact-viewed timeline write
│   │   │   └── permissions.ts        honest explanations + PermissionsAndroid requests;
│   │   │                             denial → graceful degradation dialogs
│   │   ├── screens/
│   │   │   ├── OnboardingScreen.tsx  phone auth → name → permission explanations
│   │   │   ├── HomeScreen.tsx        big SOS button, mode chip, check-in status
│   │   │   ├── ContactsScreen.tsx    add/remove contacts, linked vs SMS-only badges
│   │   │   ├── ActiveAlertScreen.tsx live status, deliberate photo capture, I'm-safe
│   │   │   ├── CheckInScreen.tsx     arm/disarm the safe-arrival dead-man's switch
│   │   │   └── IncomingAlertScreen.tsx  full-screen takeover when my network fires SOS
│   │   └── components/ui.tsx  shared Screen/Card/Button/Field vocabulary
│   │   └── services/ ... (foregroundService.ts now hosts the react-native-background-actions
│   │                        loop; batteryWatch.ts uses expo-battery; locationService.ts uses
│   │                        expo-location; permissions.ts uses expo module helpers + core
│   │                        PermissionsAndroid for SEND_SMS; smsService.ts calls the local
│   │                        OgbajeSms Expo module)
│   ├── App.tsx              root: auth gate, screen state machine, GLOBAL shake trigger,
│   │                        incoming-alert takeover, alert resume after restart
│   └── package.json         app dependencies (versions in DEPENDENCIES.md)
│
└── dashboard/               React 19 + Vite 8 web dashboard (TypeScript)
    ├── index.html           entry page (Vite serves /src/main.tsx from here)
    ├── vite.config.ts       Vite + React plugin config
    ├── src/
    │   ├── main.tsx         React root; imports leaflet CSS + app CSS
    │   ├── App.tsx          parses /alert/:alertId from the URL (no router dep needed
    │   │                    for a single route); setup/no-alert notices
    │   ├── firebase.ts      web SDK init from .env.local (never committed);
    │   │                    exports `firebaseConfigured` so the UI can explain itself
    │   ├── types.ts         TS mirrors of the Firestore shapes in DATA-MODEL.md
    │   ├── index.css        dark, projector-friendly styling (header/map/side-pane grid)
    │   ├── pages/AlertView.tsx      subscribes onSnapshot to the alert doc + locations
    │   │                            + photos + timeline; renders the live response view
    │   └── components/
    │       ├── LiveMap.tsx  react-leaflet map: OSM tiles, trail polyline, accuracy
    │       │                circle, pulsing divIcon marker, auto-follow while active
    │       └── Timeline.tsx icon-coded event feed (lucide icons, per project rule)
    ├── .env.example         names of the Firebase web-config variables
    └── package.json
```

Items marked **[planned]** are created in their build phase; the marker is removed once the
file exists. Everything else exists now.
